# uni-app 静态页面
构建贴合当前项目结构、视觉完成度高、并且便于后续接入真实数据的 uni-app 静态页面。

## 工作流
1. 先检查当前项目约定。
   - 优先阅读 `pages.json`、`App.vue`、`uni.scss`、目标页面所在目录和相邻页面。
   - 判断项目使用的是 Options API、`<script setup>` 还是混合写法，再跟随现状，不主动切换。
   - 记录公共类、主题色、图片路径约定、`uView UI` 使用方式和常见布局模式。
   - 先检查 `uView UI` 是否已有可直接覆盖当前页面结构的组件或组合方式，再检查项目内其他可复用组件。
   - 只有 `uView UI` 明显不适合当前场景时，才继续评估当前页面目录、公共组件目录和相邻业务模块里是否已有可复用组件；有合适的就直接复用，不重复新写一套。
   - 写页面前必须先检查现有公共类、公共组件和相邻页面可复用结构，不能跳过这一步直接起页面私有实现。
2. 先把需求整理成页面约束。
   - 提取页面目标、目标端、核心信息、主要区块、主 CTA、素材情况和设备优先级。
   - 如果需求不完整，可以补合理假设，但页面结构必须保持易改。
   - 文案优先写成真实占位内容，不要用 lorem ipsum。
3. 先定结构，再做视觉。
   - 先明确首屏、说明区、信任区、列表区、底部行动区等内容层级。
   - 默认按移动端优先设计，先保证常见手机宽度下自然可读、可点。
   - 通过留白、对比、卡片语言和分区节奏制造层次，不靠堆装饰救页面。
4. 用符合 uni-app 习惯的方式实现。
   - 所有页面结构都先评估 `uView UI` 组件是否可以覆盖；能覆盖就必须优先使用，不要直接手写等价结构。
   - 只有 `uView UI` 没有合适组件，或现有组件无法自然满足设计和交互要求时，才继续复用项目内其他组件或使用 `view`、`text`、`image`、`scroll-view`、`swiper` 等原生标签组合。
   - 用最简洁的代码实现页面效果，不要为了抽象而做无意义的转换、映射、包裹或中间层，避免代码过于冗余。
   - 静态页默认只做轻交互，例如切换态、弹层显隐、滚动区、轮播、锚点跳转；不要主动引入接口、全局状态或跨页业务逻辑。
5. 收尾时做移动端打磨。
   - 检查间距节奏、文本换行、图片比例、按钮热区、底部安全区和固定元素遮挡。
   - 删除无依据的预留代码、未使用引用以及和页面目标无关的占位块。
   - 如果后续大概率会接接口，确保静态结构和数据字段位置一一对应。

## 实现规则

### 项目约定优先
- 保持当前仓库的目录结构、命名方式、样式组织和组件写法。
- 不额外引入新的 UI 库、状态库或动画库，除非用户明确要求。
- 如果项目已有明确视觉语言，优先延续，不单独起一套风格。

### 页面结构
- 优先使用单文件组件写法，并与周边页面保持一致。
- 页面以展示为主时，数据结构尽量扁平，避免为了组件化而过度拆分。
- 区块命名保持清晰，方便后续替换真实文案、图片或接口字段。

### 组件与交互
- 所有页面结构都必须先评估 `uView UI` 是否已经覆盖当前场景；只要能满足需求，就必须优先使用。
- 不要先跳到本地自定义结构或手写 `view` 方案，尤其不要在 `uView UI` 可覆盖时直接自己写一套等价页面结构。
- 只有 `uView UI` 明显不适合时，才继续判断项目内是否已经有可复用的业务组件、通用卡片、弹层或列表组件；这些现有组件能够满足需求时，也必须直接复用。
- 如果 `uView UI` 和现有组件都不合适，才回退到原生标签自己搭结构。
- 以最小、最直接的实现完成目标，不为了一点点复用预期提前堆叠复杂封装。
- 交互保持轻量；如果用户请求已经超出静态页面范围，要明确指出边界变化。

### 样式与适配
- 默认使用 `scss`，优先复用现有变量、混入和公共类。
- 优先使用现有公共类；如果缺少某类样式，并且预期会在多个区块或多个页面重复使用，就先补成公共类，再在页面中复用。
- 能用公共类解决时，不要新增页面私有的等价样式；只有现有公共类明显不适合，才允许新增样式，并保持命名和职责清晰。
- 优先用 `flex` 完成布局；除非周边页面已经稳定使用，否则谨慎引入复杂布局技巧。
- 移动端尺寸优先使用 `rpx`；局部细边框、阴影和发丝线可按现状混用 `px`。
- 多个区块重复出现的卡片、标题、标签、按钮和间距模式，优先抽成公共类。
- 同类样式在当前页面内重复出现两次及以上时，默认提炼为公共类或复用已有公共类，而不是复制多份私有样式。

### 静态数据策略
- 非必要的动态数据直接写死即可，不要为了形式统一把明显一次性展示内容强行拆成响应式数据或配置。
- 可替换文案、图片、列表项、标签和按钮文案，优先集中到 `data`、常量或配置对象里。
- 不要出现硬编码散落；资源路径、跳转目标、默认状态、颜色值、间距值和可变文案尽量集中管理，避免散落在模板和样式里。
- 不管内容最终来自动态数据还是直接写死，文案、标签、状态值和模板内容都不要出现转义编码、乱码或 HTML 实体占位，必须保持为正常可读文本。
- 如果只是一次性展示、后续也不会复用的极小块内容，可以直接写在模板里，不必为了形式统一强行数据驱动。

### 编码与校验
- 保持源码中的用户可见文案为正常可读文本，不要把中文或其他自然语言文案改写成 `\uXXXX`、HTML 实体或其他转义占位。
- 如果终端、脚本检查、PowerShell 默认读取方式或临时校验命令出现乱码，先判断这是工具链编码问题，不要把页面源码改坏来迁就检查命令。
- 遇到“命令默认编码”和“页面源码可读性”冲突时，以页面源码质量为先：修改读取方式、临时脚本或校验命令，不修改页面文案为转义编码。
- 读取 `.vue`、`.js`、`.json` 等文本文件做检查时，优先使用显式 UTF-8 的读取方式或仓库中已有的 UTF-8 友好校验脚本。
- 交付前至少抽查一次目标源码文件本身，确认文件内看到的是正常文案，而不是乱码、转义串或实体编码。

## 边界判断
- 如果任务开始涉及接口联调、复杂表单、支付流程、登录鉴权或重状态编排，不再把它当作纯静态页面任务处理。
- 如果页面依赖设计稿、截图或现有页面复刻，先尽量对齐布局和信息层级，再微调视觉细节。
- 如果资源不全，优先保留清晰的占位结构和可替换字段，不伪造复杂业务数据。

## 交付要求
- 交付时要明确说明本次复用了哪些公共类、公共组件或相邻页面结构。
- 如果没有使用 `uView UI` 或现成公共组件，必须说明为什么它们不适合当前页面，而不是默认直接新写。
