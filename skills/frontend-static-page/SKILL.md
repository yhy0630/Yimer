# 前端静态页面

构建贴合 uni-app 项目结构、视觉完整、并且便于快速迭代的静态页面。

## 工作流程

1. 写代码前先确认 uni-app 项目约定。
   - 从仓库中判断页面目录、组件写法、样式组织、全局变量和常用 UI 模式。
   - 如果项目已经有现成的视觉语言和页面结构，优先保持一致。
   - 默认用最小实现满足需求，除非用户明确要求，否则不要主动引入接口请求、复杂状态或跨页面逻辑。

2. 先把需求整理成页面约束。
   - 提取页面目标、目标用户、主要区块、核心 CTA、所需素材和设备优先级。
   - 如果需求不完整，可以做合理假设，但页面结构必须保持易改。
   - 文案尽量具体，优先使用简短、真实的占位内容，不要用 lorem ipsum。

3. 先定页面结构，再做视觉细节。
   - 先明确内容层级，例如首屏、说明区、卡片区、列表区、数据区、底部行动区等。
   - 通过留白、对比或布局变化，让每个区块有明确区分。
   - uni-app 页面默认采用移动端优先布局，先保证常见手机宽度下的表现。

4. 用符合 uni-app 运行习惯的方式实现页面。
   - 优先使用项目中已接入的 `uView UI` 组件；只有在没有合适组件时，再退回到 `view`、`text`、`image`、`scroll-view` 等 uni-app 常用标签。
   - 颜色、间距、圆角、阴影等尽量复用现有变量、混入或公共样式。
   - 交互保持轻量。静态页面任务优先使用显隐、切换态、锚点式跳转或轻量滚动效果，不要默认增加复杂逻辑。
   - 只有在确实能减少重复时才复用本地组件。
   - 是否考虑多端兼容，以实际需求为准；如果用户没有明确要求全端一致，优先满足当前目标端的页面效果和开发效率。
   - CSS 优先提炼为公共类；凡是可复用的布局、间距、标题、卡片、按钮或状态样式，尽量写成公共类，方便后续页面复用和统一维护。
   - 不要出现硬编码；文案、颜色、间距、图片地址、状态值、配置项等能抽离的内容都应通过数据、变量、公共类、常量或配置统一管理。

5. 收尾时做页面打磨。
   - 检查间距节奏、文本行长、图片比例、文字溢出、空状态和对比度。
   - 验证窄屏和宽屏下的布局表现。
   - 删除无依据的预留代码、未使用引用，以及与页面目标无关的占位区块。

## uni-app 规则

### 项目约定优先

- 优先匹配本地框架和文件约定，不要额外引入新技术栈。
- 保持已有的命名方式、目录结构和样式组织习惯。
- 如果项目已经有设计系统，除非用户要求换风格，否则不要脱离它单独设计。

### 页面结构

- 优先使用和周边文件一致的单文件组件写法。
- 在页面和组件中，优先评估 `uView UI` 现成组件是否适合当前场景。
- 没有合适的 `uView UI` 组件时，再使用 `view`、`text`、`image` 等 uni-app 常用标签自行组织结构。
- 如果页面以展示为主，数据结构保持扁平，避免为了“组件化”过度拆分。
- 主体内容区块要清晰，方便后续插入真实接口数据或替换文案。

### 样式实现

- 优先采用当前项目已存在的 `scss` 组织方式。
- 优先复用项目内已有公共类；新增样式时，也优先按公共类思路组织，而不是写大量页面私有碎片样式。
- 默认按移动端尺寸设计，合理使用 `rpx`，必要时结合 `px` 处理细边框或阴影细节。
- 优先用 flex 完成布局，避免依赖兼容性不稳定的高级布局技巧。
- 如果任务要求兼容多端，避免使用只在 H5 表现正常、但在小程序或 app 端容易失真的写法。
- 如果多个区块出现相同的布局、标题、卡片、按钮或标签样式，优先抽成公共类再复用。
- 避免把颜色值、尺寸值、圆角值、层级值直接散落在页面里；优先复用现有变量或集中定义后再引用。

### 数据与配置

- 不要把可变文案、图片、列表内容、角标、标签、按钮文案直接硬写在模板结构里。
- 展示型页面也应尽量把可替换内容整理为数据对象、数组或配置项，方便后续接接口或统一修改。
- 资源路径、跳转目标、默认状态和值映射应集中管理，避免在多个区块重复硬写。

### 资源与兼容

- 图片路径、静态资源引用方式要遵守项目现有约定。
- 如果页面依赖背景图、装饰图或图标，优先使用仓库里已有资源；没有时再用清晰占位结构预留。
- 如果某个视觉效果可能存在多端差异，要优先选择更稳妥的实现方式。

## 视觉规则

- 避免通用模板感。先确定一个明确的视觉方向，再一致地执行。
- 先用字体、间距和对比建立层级，再考虑装饰元素。
- 重复使用少量视觉母题，例如强调色、边框样式、卡片语言、背景纹理或分区方式。
- 页面必须方便后续修改，不要把关键内容埋进过深的结构里。
- 如果没有品牌规范，优先使用克制但有判断的配色，不要默认蓝紫渐变模板风。
- 信息密度要适配手机阅读，避免首屏堆太多内容导致压迫感。
- 卡片、按钮、宫格和底部行动区要优先考虑手指点击区域。

## 交付规则

- 交付时说明影响页面结构的关键假设。
- 明确指出主文件，以及最值得优先调整的文案区或样式区。
- 如果需求实际上更适合动态数据、表单或应用逻辑，应在扩展范围前先明确说明。
- 如果页面后续大概率会接接口，尽量让静态结构和数据字段位置一一对应，方便后续替换。
